# /bug-fix — バグ修正

ユーザーが「バグがあります」「〜が動きません」「Issue #N のバグを直して」と言ったらこの手順に従う。

## 入力

- バグの症状（再現手順 / 期待結果 / 実際結果）
- ログ / エラーメッセージ / スクリーンショット

## 4 段階デバッグ（対症療法は禁止）

### 1. 症状の正確な記録

- エラーメッセージ全文を取得
- ログ（logcat / Xcode console / Metro）を確認
- 再現手順を 3 行以内で書く

### 2. 仮説の列挙（最低 3 つ）

- 「これが原因かも」を 3 つ以上挙げる
- 各仮説を **可能性の高い順** に並べる

### 3. 仮説の検証

- 最も可能性が高いものから検証
- ログ追加 / Jest テストで再現 / 関連コード読込

### 4. 根本原因の特定と修正

- **応急処置ではなく本質的な原因**を直す
- バグを再現する Jest テストを追加（再発防止）
- 修正実装
- `pnpm verify` 全パス確認
- リグレッションがないか確認

## 深刻度の判定

- **P0**（即修正）: クラッシュ / データ損失 / 課金壊れ
- **P1**（当日修正）: 主要機能が使えない
- **P2**（次の PR で修正）: 回避可能な不具合
- **P3**（Issue に記録）: 軽微な表示崩れ

## 制約

- 「動いたから OK」ではなく「**なぜ動いたか**」を説明できること
- 修正範囲を最小限に保つ（影響範囲を広げない）
- 受容する場合は ADR で明文化する
- 同じバグが再発しないように **必ずテストを追加** する

## 完了基準

- [ ] 根本原因が特定された
- [ ] バグを再現するテストが追加された
- [ ] 修正後にテストがパス
- [ ] `pnpm verify` 全パス
